1. Definition

본업은 UI가 아니라 experiment instrument

UI는 조작 표면이고, 본체는 reproducible experiment pipeline과 memory-annotated IR이다.

workloadmodel, serving mode, context, TTT-class behavior
hw configmemory tier, capacity, bandwidth, tiling strategy
traceOTel, Chakra, ServingSim output
metriccapacity pressure, traffic, latency, movement bytes
projectionDB row, comparable view, device-roadmap evidence
확인된 범위에서의 가설: 기존 memory-centric 연구는 architecture를 주장하고, 기존 DSE tool은 fixed device class 안에서 optimize한다. 여기서 노리는 빈자리는 device class 미정 상태에서 축을 흔들며 가설을 싸게 실험하는 instrument다.
positioning: solver가 아니라 instrument. 자동 최적해 탐색이 아니라 domain expert가 memory/workload axis를 움직이고 evidence를 쌓는 explorer's instrument.

2. IR / Schema

Metric causal chain에 걸리는 것만 1급 field

schema는 빼는 게 지옥이다. 초기에는 causal field와 opaque attributes를 분리한다.

node / op

input tensor refs output tensor refs working set

op 생존 중 필요한 working set을 capacity pressure의 최소 근거로 둔다.

tensor

size dtype allocated_at freed_at residency tiling strategy id

movement edge

src tier dst tier bytes sync / async

data movement를 first-class edge로 둬야 traffic과 latency의 causal chain을 잃지 않는다.

Fill level 의미 첫 사용
L0 op-level graph + tensor size/lifetime capacity peak, rough traffic, ServingSim first projection
L1 memory tier residency + per-tier movement bytes partitioning/tiling이 metric에 미치는 영향 가시화
L2 kernel-level tiling schedule, intra-kernel reuse, hw-optimal runtime logic SynTorch가 vLLM에 붙은 뒤

L0/L1/L2는 서로 다른 schema가 아니라 같은 schema의 fill level이다. source adapter가 못 채우는 값은 명시적 null로 둔다.

3. Critical Path

지금 필요한 것은 first projection과 trust ladder

source-agnostic control plane을 먼저 세우고, SynTorch와 TTT를 같은 IR로 합류시킨다.

지금
SynTorch ↔ A100 / OTel 검증 golden test + L0 IR paper validation. P1 생존조건이자 control plane 척추.
병렬
좁은 keyword radar를 시작해 DeepStack, Minsoo Rhu, TTT writeback traffic 누락 위험을 줄인다.
다음
추론: ServingSim을 source A로 먼저 연결해 source-agnostic first projection을 만든다.
그 후
SynTorch를 source B로 합류시키고, TTT-class workload를 future writeback-memory axis로 추가한다.
Open decision: 이번 주 착수는 (a) 좁은 radar 먼저 vs (b) SynTorch 검증 golden test 먼저. 둘 다 급함. "안 돌면 더 불안한 쪽"으로 선택. 이 webapp은 자동 선택하지 않는다.

4. Paper / Patent Pipeline

P1 도구 → P2 방법론 → P3 미래 축

P3는 출력이지 입력이 아니다. P1/P2의 신뢰사다리 없이 TTT paper를 단정하지 않는다.

P1 · Tool

SynTorch: An Executable Synthetic Frontend for Memory-Centric Design Space Exploration of Unbuilt AI Hardware

PyTorch-like frontend와 abstract backend로 unbuilt memory system 위에서 hardware-optimal Chakra trace를 만든다.

Venue: ASPLOS, MICRO, MLSys workshop fallback

Patent: virtual memory hierarchy 위 executable trace 생성 + capacity/traffic 유도.

Survival: A100 / OTel 대비 trace validation.

P2 · Method

Tracking the Moving Axes: Evidence-Based Projection of Memory Demand Shifts in Evolving AI Workloads

OTel, SynTorch, ServingSim/ASTRA-sim을 하나의 memory-annotated IR로 통합해 capacity vs bandwidth 비율의 이동을 추적한다.

Venue: ISCA vision/projection, IEEE Micro, HotInfra

Patent: heterogeneous trace 통합 IR + dependent-variable ratio movement detection.

P3 · Future Axis

When Inference Writes Back: The Memory Traffic Signature of Test-Time Training and Its Architectural Implications

TTT는 inference 중 weight update를 만들며 gradient, optimizer state, write traffic, updated weight reuse라는 새 memory axis를 만든다.

Venue: HotChips, HotInfra, ASPLOS WACI, later MICRO/ISCA

Patent: inference write traffic을 겨냥한 memory device structure.
