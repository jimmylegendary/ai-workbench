# RB-003: Implement the file store, the entity schemas, and the wbtraffic.v0 schema

- Status: ready
- Phase: phase-0-foundations
- Depends on: [RB-000, RB-001, RB-002]
- Implements design: [../../04-data-layer/data-model.md](../../04-data-layer/data-model.md), [../../04-data-layer/storage-and-scheduling.md](../../04-data-layer/storage-and-scheduling.md), [../../01-decisions/ADR-0002-hypothesis-representation.md](../../01-decisions/ADR-0002-hypothesis-representation.md), [../../01-decisions/ADR-0003-experiment-ledger.md](../../01-decisions/ADR-0003-experiment-ledger.md), [../../01-decisions/ADR-0004-writeback-traffic-schema.md](../../01-decisions/ADR-0004-writeback-traffic-schema.md), [../../01-decisions/ADR-0006-implication-mapping.md](../../01-decisions/ADR-0006-implication-mapping.md), [../../01-decisions/ADR-0007-storage-and-scheduling.md](../../01-decisions/ADR-0007-storage-and-scheduling.md)
- Produces: the shared record envelope + the entity schemas (`Source`, `Claim`, `Hypothesis`, `ExperimentEntry`+`Result`, `ImplicationMap`, `WritebackTrafficSchema`/`wbtraffic.v0`, `ExportBundle`) with validators enforcing the data-model §5 invariants; the append-only file store reader/writer with monotonic IDs and a "current" resolver; the `wbtraffic.v0` schema with all-numerics-default-`null` + `basis` modeled-vs-measured.

## Objective
Give CAW-06 its OWN persistent data layer and the typed records every later stage reads/writes. "Done" = each entity from data-model.md §3 has a schema model (pydantic or JSON Schema) carrying the shared envelope (§2) and validated against the §5 invariants; the store writes one markdown/JSON record per entity into the correct `store/<dir>/`, assigns stable monotonic IDs, is append-only with supersede, references large artifacts by path only, and exposes a "current"-state resolver over append-only records; and the `wbtraffic.v0` schema exists with every numeric defaulting to `null` and a required `basis: modeled|measured` flag. The validators must make the no-overclaim invariants impossible to violate by construction: no Hypothesis-bearing record without `status`+`confidence`; `generated` content is `evidence:false` and can never promote a status or be an `evidence_ref`; every `wbtraffic.v0` numeric is `null` or sourced, never invented. This is schema + store only; no pipeline stage logic.

## Preconditions
- [ ] RB-002 complete: ports/registry/fakes exist; `check` is green.
- [ ] Schema-validation tool chosen (resolving tech-stack `TODO`: pydantic v2 vs jsonschema). Record the pin.
- [ ] Entity names and ID prefixes from data-model.md / repo-structure.md are fixed: `SRC/CLAIM/HYP/EXP/IMAP/WBT/EXB` (use exactly; do not rename).

## Steps

1. **Do:** Implement the shared envelope (data-model.md §2) as a base model: `id`, `kind`, `created` (leave as a real timestamp at write time — do NOT invent dates in fixtures), `provenance{source_ids, origin, retrieved_at}`, `boundary` (`internal|export:caw-01|export:caw-02`), `status`, `lineage{supersedes, derived_from}`. A `generated` flag/`evidence:false` marker is representable on any sub-field.
   **Verify:** a record missing any required envelope field is rejected by the validator; a unit test asserts rejection.

2. **Do:** Implement `schemas/source.py` (data-model §3.1): `Source` with `title`, `authors`, `canonical_id`, `versions`, multi-entry `provenance` (incl. a CAW-05 entry flagged `evidence:false`), `boundary: internal`. Rediscovery merges into one `Source` with multiple provenance entries (dedup performed by the pipeline later, not here).
   **Verify:** a `Source` with two provenance origins validates; a CAW-05 provenance entry carries `evidence:false`.

3. **Do:** Implement `schemas/claim.py` (§3.2): `Claim` with `source_id`, `statement`, required `evidence_span` (verbatim) + `source_locator`, `claim_type` enum, `writes_back: true|false|unknown` (default `unknown`), `asserted_by`, and `status` fixed to `unverified`. Validator forbids `status=supported` on a Claim (extraction never emits supported).
   **Verify:** a Claim without `evidence_span` is rejected; setting `status=supported` is rejected; default `writes_back` is `unknown`.

4. **Do:** Implement `schemas/hypothesis.py` (§3.3): `Hypothesis` with `statement`, `from_claims`, `status` (`hypothesis|supported|refuted|inconclusive`, default+floor `hypothesis`), `confidence` (5-value, default `very-low`), `evidence_strength`, `agreement`, optional `likelihood`, `falsifiability` (required to leave `hypothesis`), `reproducibility`, `evidence_ids`, and an append-only `status_log`. Validator: never serialize without `status`+`confidence`; **calibrated uncertainty with a HARD evidence cap** — generated/`evidence:false` evidence can NEVER raise `status` or `confidence` (ADR-0002 §4). Leaving `hypothesis` requires `falsifiability` set.
   **Verify:** serializing a Hypothesis without `status` or `confidence` is rejected; a status promotion driven only by generated evidence is rejected; moving off `hypothesis` without `falsifiability` is rejected; `status_log` is append-only (no in-place edit).

5. **Do:** Implement `schemas/ledger_entry.py` (§3.4/§3.5): `ExperimentEntry` with `hypothesis_id`, `claim_ref`, `status` (`planned|running|done|aborted`), pre-registered `prediction{metric, baseline, expected_direction, decision_rule}`, and the **reproducibility gate** block `repro{config_path, seeds, code_rev, data_ref, env_lock, hardware, budget}`; an embedded `Result{verdict: supported|refuted|inconclusive|invalid, metrics_path, observed_effect, negative_result, failure_mode}`; and the optional `writeback_observed` hook (a MEASURED number; `bytes_per_update` null until measured). Validators: a Result missing config+seed+env in `repro` is flagged **non-reproducible → not usable as evidence** (the gate the ledger writer owns, not the runner); `invalid` is distinct from `refuted`; negative results (`refuted`/`inconclusive`/non-null `failure_mode`) are retained and marked, never dropped.
   **Verify:** an entry without seeds/env in `repro` is marked non-reproducible and refused as evidence; an `invalid` verdict does not count as `refuted`; a `negative_result=true` entry validates and is retained (failures useful).

6. **Do:** Implement `schemas/implication_map.py` (§3.6): `ImplicationMap` with `finding_ref`, a `summary` **forced `evidence:false` (generated, never an evidence_ref)**, and `implications[]` each with `domain` (fixed 6-domain enum), `statement`, `status`, `confidence` (3-value), `evidence_refs` that MUST resolve to a `Result` or `Claim` (never the summary), optional `writeback_payload_ref`, and `export_targets` (routing hint only).
   **Verify:** an `evidence_ref` pointing at the summary is rejected; the `summary` is always `evidence:false`; a `domain` outside the enum is rejected.

7. **Do:** Implement `schemas/wbtraffic_v0.py` (§3.7, ADR-0004): `WritebackTrafficSchema` with `schema_version: "wbtraffic.v0"`, `ttt_variant`, `provenance`, mandatory `uncertainty{status, confidence}` (an ADR-0002 status), required `basis: modeled|measured`, and the field groups `fast_weights`, `update`, `writeback`, `ratio_curve`, `assumptions`, `open_questions` — **every numeric defaults to `null`**. Validator: a non-null numeric without either `basis=measured` (sourced from a ledger Result) or an entry in `assumptions` (if `basis=modeled`) is rejected — **no invented numbers**; a `modeled` value with no assumptions is rejected; `uncertainty` is mandatory.
   **Verify:** all numerics default `null`; setting a numeric with `basis=modeled` but no assumption is rejected; a record missing `uncertainty` or `basis` is rejected; `open_questions` is representable and non-stripped.

8. **Do:** Implement `schemas/export_bundle.py` (§3.8): `ExportBundle` with `target: caw-01|caw-02`, semver `schema_version` (in-band, no shared registry), `producer: "caw-06"`, `content_hash`, `provenance`, a target-specific `payload`, and a `receipt{emitted_at, result: ok|rejected, reason}` (a rejected export stays exportable). Validator: nothing crosses `boundary` stripped of `status`/`uncertainty`; the bundle is self-describing (no import of a shared schema registry).
   **Verify:** a bundle whose payload drops the carried status/uncertainty is rejected; `content_hash` enables idempotent upsert (same id+hash = no duplicate); a `rejected` receipt is still a valid, retained record.

9. **Do:** Implement the store reader/writer in `core/store.py` (storage-and-scheduling.md §2/§3): write one md/JSON record per entity into the correct `store/<dir>/`; assign stable monotonic IDs per prefix (never reused; superseded records keep their ID + gain `lineage.supersedes`); append-only with supersede (correction = new record/`StatusEvent`, never in-place edit); large artifacts referenced by path under `artifacts/EXP-XXXX/` only (never inlined). Wire it to validate every record against its schema before writing.
   **Verify:** writing then reading each entity round-trips; ID assignment is monotonic and gap-tolerant; superseding produces a new record retaining the old on disk; writing an invalid record (failing step 1–8 validators) is refused before touching disk.

10. **Do:** Implement the "current"-state resolver in `core/resolver.py` (storage-and-scheduling.md §3): pure function over `store/` computing "current status" per Hypothesis (newest `StatusEvent`) and "current verdict" per Hypothesis (newest non-superseded `ExperimentEntry`). Deleting `store/index/` must lose nothing (index is disposable/rebuildable).
   **Verify:** with two superseding entries, the resolver returns the newest; with an appended `StatusEvent`, "current status" reflects it; deleting `store/index/` and re-resolving yields identical results.

## Acceptance criteria
- [ ] All eight entity schemas (`Source`, `Claim`, `Hypothesis`, `ExperimentEntry`+`Result`, `ImplicationMap`, `WritebackTrafficSchema`, `ExportBundle`) exist with the shared envelope and pass `typecheck` — names/prefixes match data-model.md exactly (P0 exit: "every record kind has a schema + validator").
- [ ] The data-model §5 invariants are validator-enforced and tested: no Hypothesis-bearing record without `status`+`confidence`; **HARD evidence cap** (generated evidence never promotes status/confidence); `Claim.asserted_by` present and never restated as our conclusion; every `wbtraffic.v0` numeric `null` or sourced (never invented); nothing crosses a `boundary` stripped of status/uncertainty.
- [ ] The reproducibility gate is enforced by the store/ledger writer: an entry without config+seed+env is marked non-reproducible and refused as evidence; `invalid` ≠ `refuted`; negative results are retained, classified, and surfaceable (failures useful).
- [ ] `wbtraffic.v0` carries `schema_version`, `uncertainty`, `basis: modeled|measured`, all-numerics-default-`null`, `assumptions`, and `open_questions`; modeled numbers require assumptions.
- [ ] The file store writes one record per entity into the correct `store/<dir>/` with monotonic non-reused IDs, append-only+supersede, artifacts by path only; round-trips read/write; refuses invalid records before writing.
- [ ] The "current"-state resolver returns newest status/verdict over append-only records; deleting `store/index/` loses nothing (P0 exit: "store dirs create/round-trip"). Tree stays green.

## Rollback / safety
- All changes are additive (schema modules, `core/store.py`, `core/resolver.py`, tests). Rollback = revert the `schemas/*.py` and the two core modules to RB-000 placeholders.
- Safety: the store is append-only — rollback must NEVER delete real records under `store/` (failures are first-class, ADR-0007). A mid-way failure leaves any partially written record refused by validation before it reaches disk, so the store never holds an invalid record. Use atomic write (temp file + rename) so an interrupted write cannot corrupt a record.

## Hand-off
The next runbooks (Phase 1+) can assume: typed, validated records for every entity; an append-only file store with monotonic IDs, supersede, artifacts-by-path, and a current-state resolver; and a `wbtraffic.v0` schema ready to be filled by the analytic L0 estimator in P3. With the schemas, store, ports (RB-002), boundary rule + op-manifest (RB-001), and tree (RB-000) in place, P0 is complete: the ingestion pipeline (RB-1XX) can persist its first real Source/Claim/Hypothesis, and every gate (status/uncertainty + evidence cap, reproducibility, export-eligibility) is enforced by schema + store, not by any adapter.
