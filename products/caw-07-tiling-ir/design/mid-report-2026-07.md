# CAW-07 중간 보고 (2026-07)

추상화된 타일링/병렬화 IR — 목표·커버리지 재정의, 메모리 기능 판단, 그리고
"이 IR로 전체 시뮬레이션을 돌리면 무엇이 이득이고 회사 의사결정에 왜 중요한가".

---

## 1. 우리가 만드는 것과 커버리지 (재브리핑)

**비전.** 사용자가 CAW-01에서 **아직 존재하지 않는 새 HW(GPU형/NPU형)** 와 **새 병렬화/
타일링**을 설계하고, **vLLM + syntorch + hw**로 **엔드투엔드 시뮬레이션**을 돌린다.
CAW-07 = 이걸 가능케 하는 **분석적으로 비용이 매겨지는(analytical), 코드젠 없는 타일링/
병렬화 IR** — 하나의 HW-파라메트릭 매핑이 **임의의 twin에서 자동으로 re-tile + re-cost**.

**커버리지 목표 (되게 하려는 것).**
- 임의의 메모리+연산 계층(tray/package/die/component → LevelStack) 위에서 타일링/병렬화를 **표현**.
- **결정-등급(decision-grade)의 상대적 비용**(traffic/roofline) — 독립 오라클로 검증된 신뢰도.
- syntorch/vLLM에 **읽기전용으로 부착**해 authored HW+tiling에서 **엔드투엔드 서빙 수치**.
- intra-device 타일 ↔ inter-device mesh ↔ network 를 하나의 IR로 (후속, ASTRA-sim).

**커버리지 비목표 (명시적으로 안 하는 것).**
- 컴파일러/코드젠 아님. novel HW에서 **기능적 실행/사이클-정확 시뮬 아님**.
- **절대 실리콘-정확 수치 아님.** 이건 **탐색(what-if)·상대비교** 도구다.

**현재 위치.** P2(신뢰 가능한 cost + 실제 NPU twin + 검증)에서:
traffic 모델이 **ZigZag와 바이트 단위 일치**(spill 포함) + 물리 하한 + TPU-v1 ridge 재현.
설계/서베이/논문 완료. syntorch = 팀용 런북(RB-01). P3(팀)·P4·P5·P6 남음.

## 2. 메모리 마이크로아키텍처 기능 — 목표 기준 판단

판단 기준(=우리 커버리지 목표): **"이 요소가 *설계 비교의 승패*를 바꾸는가, 탐색 범위 안에서?"**
완전성(HW 백과사전)이 아니라 **결정을 뒤집는 소수**만 우리 경로에 올린다.

- **Tier A — 하되 얇게 (결정을 뒤집음, 비용 저렴).** ports(read/write 분리 + shared flag),
  double-buffer flag + overlap α, spatial-utilization 페널티.
  *왜:* compute/memory-bound 판정과 "내 버퍼가 partial-sum RMW를 숨기나"를 바꿈 = 아키텍트가
  매일 묻는 질문. (이미 보유: capacity, per-operand placement, reduction-combine, spill.)
- **Tier B — 수요 기반 (결정이 걸릴 때만).** line/burst granularity, bank_count(+η_bank),
  η_dram, channel 명시(×BW), device-kind 프리셋. *왜:* 2차 derate — 어떤 결정이 여기 걸릴 때만 값어치.
- **Tier C — 새 설계 축이 필요할 때만 (큰 작업).** managed-vs-cache 태그 + cache hit-rate 경로;
  energy/area(ERT/ART). *왜:* cache는 우리 folding의 정확성 근거(결정적 residency)를 깨므로
  **별도 plug-in**으로만; 에너지 목표가 실제로 생기면.
- **Tier D — 범위 밖, 명시적 hand-off (절대 twin에 가짜로 넣지 말 것).** DRAM command timing/
  refresh/queueing, 정확한 bank conflict, NoC 혼잡, 다중엔진 SoC 스케줄. *왜:* 런타임 접근순서·
  다중주체 창발 → closed-form 없음. 가짜로 넣으면 **false precision = 신뢰 훼손**. → Ramulator/
  ASTRA-sim/gem5로 넘기고 우리는 컴포넌트별 분석값만 노출.

**결론(판단).** "반드시 다 커버"가 아니다. **Tier A만 결정-등급 경로의 필수**, B는 온디맨드,
C는 축이 생길 때, **D는 분석적으로 안 함**. 이 규율이 우리를 정직하고 목표(상대비교)에 집중하게 함.
→ 상세 근거·요소별 표: [memory-microarchitecture-in-ir.md](memory-microarchitecture-in-ir.md).

## 3. 이 IR + syntorch + 풀-시뮬의 이득 (없이 돌린 것 대비)

**"이 IR 없이"는 무엇인가 (대안들의 한계):**
- (a) 칩/RTL/사이클 시뮬 제작 → 수개월~수년 + 큰 NRE, **설계를 커밋한 뒤에야** 가능, 폭넓은 탐색 불가.
- (b) 손으로 만든 스프레드시트/일회성 스크립트 → 설계마다 재작성, **비교 불가(bespoke)**, 실제
  워크로드(vLLM)와 분리, 오류 잦음.
- (c) 기존 GPU에서 vLLM 실행 → **당신의 새 HW에 대해선 아무것도 말해주지 않음.**
- (d) novel HW의 풀 사이클 시뮬 → **novel HW엔 기능 백엔드가 없음**(바로 그 문제) + 탐색엔 너무 느림.

**CAW-07의 이득 (= 위 대안들이 못 주는 델타):**
1. **아무것도 만들기 전에 HW+타일링 설계를 탐색.** 임의의 새 메모리/연산 계층 + 새 타일링/병렬화를
   **실제 워크로드(vLLM prefill/decode)** 에 대해 **초 단위·분석적으로** 평가. (a)~(d) 어느 것도 못 줌.
2. **동일 워크로드, 다수 설계의 사과-대-사과 비교.** IR 하나·cost 엔진 하나, twin/매핑만 교체 → **직접 비교 가능**.
3. **실제 서빙 워크로드에 결부.** syntorch/vLLM에 붙으므로 수치가 *당신의 실제 모델+서빙 패턴*
   (배치·KV·decode)에 대한 것 — 합성 마이크로벤치가 아니라 **진짜 애플리케이션 기준의 설계 결정.**
4. **알고리즘과 하드웨어의 co-design.** 타일링/병렬화가 같은 IR에 있으니 "SRAM을 늘리면/dual-port면/
   reduce 가능한 NoC면, 내 새 타일링이 실제로 그걸 활용하나?"를 물을 수 있음 — **가속기 설계의 진짜 이득이 나오는 루프.**

**아키텍처 설계에 미치는 영향 & 중요도.**
- 아키텍처 결정은 **고비용·비가역**(테이프아웃 후 되돌릴 수 없음). 메모리 계층/포트/병렬화의 잘못된
  선택 = **칩 한 세대(≈수천만~수억 달러, 1~2년) 손실.**
- 이 결정의 **가장 큰 지렛대가 메모리 계층 + 타일 매핑**이다 — 우리 자체 결과가 증명: *같은 GEMM*이 한
  twin에선 821µs **memory-bound**, 다른 twin에선 **compute-bound**. 어느 후보 설계가 이기는지의
  판정(compute/memory-bound + traffic)을 **초기에** 주는 게 정확히 아키텍트가 필요로 하는 신호.
- 즉 "있으면 좋은 분석"이 아니라 **제품에서 가장 비싸고 가장 비가역인 결정을 디리스킹**하고,
  **탐색 루프를 수개월 → 수분으로** 압축하는 것. 이것이 중요도의 본질.

## 4. 이해관계자별 어필 & 회사 의사결정 구조에서의 의미

같은 하나의 분석 결과를, 각 층이 **다른 언어**로 읽는다 — 그래서 조직적으로 유용하다.

**경영진(C-level) — 언어: 리스크·출시속도·자본효율·차별화.**
- "새 가속기 설계를 **칩 NRE를 쓰기 전에 실제 워크로드로 평가** → 수천만 달러 베팅을 디리스킹."
- "실리콘 이전 탐색 루프를 **수개월 → 수분** → 단위시간당 설계 반복↑ = 더 나은 칩 + 빠른 출시."
- "**HW/SW co-design → 우리 서빙 워크로드에 맞춘 방어 가능한 차별화** (범용 칩이 아님)."
- "공개 분석 기반(Timeloop/ZigZag로 검증)을 재사용 → **신뢰성 있고 저비용**, 과학 프로젝트 아님."
- 한 줄: **잘못된 실리콘 베팅↓ + 반복↑ + 워크로드-맞춤 차별화.**

**실무 엔지니어(HW/perf/ML-sys) — 언어: 구체성·실행가능·신뢰.**
- "twin+타일링을 몇 줄로 authoring → traffic/roofline bound/remainder/어디서 spill 하는지를 **초 단위,
  ZigZag로 바이트 검증**된 값으로."
- "매일의 질문에 답: 이 레이어가 이 설계에서 memory/compute-bound인가? SRAM 늘리면 도움되나?
  dual-port가 누산기 RMW를 숨기나? split-K가 combine 비용을 무나?"
- "이 IR이 **나중에 런타임 팀에 넘길 매핑 그 자체** → 버려지는 분석이 아님."
- 가치: 실제 워크로드로 말하는 **빠르고 신뢰 가능한 what-if 샌드박스**(스프레드시트 대체).

**제품 리더(설계·로드맵 오너) — 언어: 커버리지·확신·우선순위·조율.**
- "경쟁하는 HW 제안들 사이를 **비교 가능·방어 가능**하게 선택 — 가정이 명시되고 검증 게이트가 문서화
  (무엇이 탐색-등급인지 정직)."
- "결정을 뒤집는 요인(우리가 모델링) vs 시뮬 필요 요인(명시적 hand-off)을 **분리** → 팀이 각 수치를
  얼마나 믿을지, 어디서 사이클 시뮬로 escalate 할지 정확히 앎."
- "HW 로드맵을 SW/서빙 로드맵과 결합(co-design) → **두 팀이 공유하는 하나의 아티팩트.**"
- 가치: HW/SW를 정렬하고 설계 선택을 위로 방어할 수 있게 하는 **공유되고 정직한 의사결정 도구.**

**의사결정 구조에서의 의미.**
결정은 흐른다: **엔지니어 제안 → 리더 우선순위/선택 → 경영진 펀딩/커밋.**
각 층은 *같은 분석 결과*의 다른 슬라이스가 필요하다 — 엔지니어는 원시 what-if, 리더는 비교가능성+신뢰
구간, 경영진은 리스크/시간/자본 프레이밍. CAW-07은 **하나의 분석 결과를 세 층이 각자의 언어로 읽게** 하여
**회사가 하는 가장 비싼 결정에 대해 세 층을 동일 증거로 정렬**한다 — 이것이 조직적 가치.

**신뢰 유지를 위한 필수 단서.** 이건 **결정-등급 *상대* 값이지 실리콘-*절대* 예측이 아니다.**
"디리스크·우선순위·탐색"으로 팔아야지 "정확한 TFLOPS 예측"으로 팔면, 그 수치를 검증할 엔지니어에게서
신뢰를 잃는다. (그래서 우리는 검증 게이트와 Tier D hand-off를 명시한다.)

## 5. 정직한 상태 — 가치가 "실현"되려면

- **오늘 실증된 것:** cost/traffic 코어가 검증됨(P2) — 임의 twin에서 **연산자별 바이트-정확 what-if**.
- **아직인 것(= "풀 시뮬을 돌린다"의 실현):** syntorch 부착(P3, **팀**) + 서빙 루프(P6)가 있어야
  "이 새 칩의 tokens/sec" 수준의 엔드투엔드 가치가 나온다. **가치 제안은 신뢰 가능·궤도 위이나
  엔드투엔드로는 아직 미실현.**
- **따라서 우선순위(제 판단):** P4(transform-dialect식 스케줄 프론트, 다중레벨/fusion) →
  P5(three-tier + ASTRA-sim) → P6(vLLM serving loop) 이 "풀 시뮬 가치"의 임계경로.
  Tier A 메모리 기능(ports/double-buffer/overlap/util)은 그 과정에서 **결정-등급을 강화**하는 저비용 추가.
